**ph_store_intake_results(owner_email, mode, intake_data, project_name=None) -> dict:**
- Create IntakeSession with the provided fields
- db.add, db.commit, db.refresh
- Return dict with session_id, owner_email, mode, project_name, status, created_at

**ph_get_intake_results(session_id) -> dict:**
- Query IntakeSession by id (parse session_id as UUID)
- If not found: return {"error": f"Session {session_id} not found"}
- Return dict with all fields including intake_data

**ph_list_intake_sessions(owner_email, status=None) -> list[dict]:**
- Query IntakeSession filtered by owner_email
- If status provided, also filter by status
- Order by created_at DESC
- Return list of dicts with session_id, mode, project_name, status, created_at

**ph_sync_manifest(project_id, manifest_yaml) -> dict:**
- Query Project by id
- If not found: return {"error": f"Project {project_id} not found"}
- Parse manifest_yaml with yaml.safe_load to get parsed_data
- Query existing Manifest for this project_id
- If exists: update raw_yaml, parsed_data, fetched_at=now, sync_source="mcp"
- If not exists: create new Manifest with project_id, raw_yaml=manifest_yaml, parsed_data, sync_source="mcp"
- Commit and return dict with manifest_id, project_id, synced_at

The Manifest model already has the `sync_source` column (added in Plan 01, Task 1). This tool sets it to "mcp" to distinguish MCP-pushed manifests from GitHub-refreshed manifests, preventing the circular sync pitfall (RESEARCH.md Pitfall 1).

**ph_record_express_run(owner_email, base_ci=None, automated=False) -> dict:**
- Create ExpressMetric with the provided fields
- db.add, db.commit, db.refresh
- Return dict with id, owner_email, base_ci, automated, created_at

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client to portal backend | API key auth via existing ApiKeyAuth middleware. All new tools inherit protection automatically. |
| Manifest YAML input | User-provided YAML string parsed with yaml.safe_load (prevents arbitrary code execution). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Tampering | ph_sync_manifest YAML input | mitigate | Use `yaml.safe_load()` (not `yaml.load()`) to parse manifest YAML. safe_load only allows basic Python types, preventing arbitrary object construction. |
| T-02-05 | Information Disclosure | ph_list_projects owner_email filter | accept | Filter is honor-system (caller provides their own email). Acceptable for internal team (<20 users, API keys are per-person). Revisit if user base grows. |
| T-02-06 | Information Disclosure | ph_list_intake_sessions owner_email filter | accept | Same honor-system pattern as ph_list_projects. Internal team only. |
| T-02-07 | Spoofing | ph_store_intake_results owner_email | accept | Caller provides email. No server-side verification that API key holder matches email. Acceptable for internal team where API keys are individually assigned. |
| T-02-08 | Elevation of Privilege | New MCP tools without auth | mitigate | All tools registered with @mcp.tool() on the same mcp instance. ApiKeyAuth middleware intercepts every on_call_tool call automatically. Verified in Phase 1 (Plan 01-03). |
</threat_model>
